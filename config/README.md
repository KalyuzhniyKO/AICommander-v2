# Конфигурация моделей

`config/models.json` — локальный файл, который не коммитится. Создайте его из одного из примеров:

```bash
cp config/models.example.json config/models.json
# или
cp config/models.free.example.json config/models.json
```

## Как выбирать модели

- Основные роли (`manager`, `architect`, `designer`, `coder`, `reviewer`) используют OpenRouter: `openrouter/<model_id>`.
- `premium_reviewer` опционален и может использовать OpenAI: `openai/<model_id>`.
- Актуальные `model_id`, цену, лимиты и доступность проверяйте в кабинете/документации OpenRouter и провайдера. Примеры в этом репозитории показывают формат, а не гарантируют бесплатность или доступность конкретной модели.

## Fallback порядок

Список моделей для роли читается сверху вниз. Backend пробует первую модель, при ошибке сохраняет понятное сообщение в SQLite и переходит к следующей. Если все модели роли недоступны, роль получает локальный fallback-ответ, а workflow не падает.

## Если модель недоступна

1. Запустите `POST /models/check` из UI или API.
2. Посмотрите `GET /models/status` и ошибки в блоке статуса моделей.
3. Проверьте `OPENROUTER_API_KEY`, лимиты, баланс и правильность `model_id`.
4. Замените недоступную модель в `config/models.json` или переместите её ниже в fallback-цепочке.

## Почему free-пример осторожный

Бесплатные модели OpenRouter могут появляться, исчезать, менять provider prefix, лимиты и доступность. Поэтому `models.free.example.json` — только шаблон структуры. Не считайте указанные значения гарантированно бесплатными: перед запуском выберите актуальные model IDs в OpenRouter.
